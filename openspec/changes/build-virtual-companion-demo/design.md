## Context

当前变更面向《今日养元》PRD，目标是在现有仓库内落地一个适合演示的虚拟陪伴单页应用。需求强调主视觉角色、健康任务、即时奖励、等级成长和轻国风疗愈风格，同时明确排除登录、后端、多页面和复杂持久化。

当前版本已经具备系统任务、摄像头验证和奖惩逻辑，但所有进度都只存在前端内存中。新增需求是把修为进度、任务完成状态和当前鼓励语持久化，并通过一个后端接口来读取和保存。

## Goals / Non-Goals

**Goals:**
- 为单页演示体验定义清晰的页面模块和状态模型。
- 明确静态任务、鼓励语和等级阈值的数据组织约束。
- 约束任务完成、鼓励语切换、养元值累计和升级提示的交互流程。
- 通过浏览器摄像头完成本地动作验证，并仅在验证成功后允许任务完成。
- 将任务包装成小说系统式任务面板，并在失败时扣除修为进度。
- 通过后端接口持久化用户当前修为、已完成任务和鼓励语进度。
- 确保实现路径简单可运行，适合本地演示。

**Non-Goals:**
- 不设计用户账号、登录注册或云端同步。
- 不设计真实任务推荐算法、个性化生成逻辑或社交能力。
- 不要求复杂动画系统、剧情系统或多页面信息架构。
- 不引入数据库、中间件或服务端渲染方案。
- 不实现高精度人体姿态识别模型或上传视频到服务端分析。

## Decisions

### 1. 使用最小 Node.js 后端统一托管静态页和 API
新增一个轻量 Node.js 服务端，既负责提供静态资源，也提供读取/保存进度的 JSON API。这样避免前端直接面对 `file://` 场景下的跨域和摄像头安全上下文问题，同时保持部署简单。

### 2. 进度存储采用本地 JSON 文件
后端将用户进度保存到项目目录下的数据文件，例如 `data/progress.json`。这足够满足单用户本地演示需求，也便于快速重置和观察数据变化。

### 3. 持久化接口采用单记录读写模型
提供 `GET /api/progress` 读取当前进度，`POST /api/progress` 覆盖保存当前进度。由于本项目没有账号体系，后端只维护一个默认用户记录，避免无谓复杂度。

### 4. 前端初始化时先拉取后端进度
页面启动后先请求后端进度，再把已完成任务、当前修为和鼓励语游标恢复到本地状态，随后再渲染主界面。若后端无记录，则使用默认初始状态。

### 5. 关键状态变化后自动保存
任务完成、任务失败、鼓励语切换等会影响用户状态的行为，应在前端更新本地状态后异步调用保存接口。若保存失败，页面保留当前交互结果，但通过提示告知用户本次进度未落盘。

### 6. 后端保存时做最小字段白名单校验
后端只接受 `score`、`encouragementIndex`、`completedTaskIds` 和必要元数据，避免把完整前端运行态原样写入磁盘，减少脏数据风险。

## Risks / Trade-offs

- [本地 JSON 文件仅适合单用户演示] → 不适合多用户并发，但足够满足当前原型目标。
- [前端保存失败时会与服务端状态不一致] → 通过错误提示提醒用户刷新前注意本次进度未持久化。
- [统一由本地 Node 服务提供静态资源] → 需要从 `node server.js` 启动页面，而不是双击 HTML 文件；但这样也正好满足摄像头安全上下文需求。
- [无数据库] → 重启服务不会丢失，但不具备复杂查询和并发控制能力；这是当前阶段的有意取舍。
